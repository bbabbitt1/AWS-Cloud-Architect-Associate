Storage Extras: Advanced Migration & File System Options

Overview

This page covers advanced AWS storage services for data migration, edge computing, high-performance file systems, hybrid cloud access, and managed file transfer. These services are critical for exam scenarios involving large-scale data movement, on-premises integration, and specialized workloads.

AWS Snowball & Snowball Edge

Snowball devices are highly secure, portable appliances for collecting, processing, and migrating data into and out of AWS. Use Snowball when network transfer would take more than a week.

Snowball Data Migration Timeline

Comparison of network transfer times vs Snowball transfer times:

Exam tip: If network transfer takes more than a week, Snowball is the answer. For PB-scale, Snowball wins hands down.

Snowball Device Types

  • AWS Snowball (Storage Optimized): Up to 80 TB capacity, primarily for data migration.
  • AWS Snowball Edge (Compute & Storage): Up to 100 TB storage + local compute (EC2 instances or Lambda functions). Used for edge computing scenarios.

Edge Computing with Snowball Edge

Process data while it's being created on remote/edge locations without real-time internet connectivity:

  • Examples: IoT sensor preprocessing at remote oil rigs, video transcoding at film studios with limited bandwidth, machine learning inference at retail stores, pre-processing sensor data from ships or aircraft.
  • Launch EC2 instances or Lambda functions directly on Snowball Edge devices to run workloads locally.
  • Use case: Collect + preprocess data locally, then ship the device to AWS for final ingestion and centralized analytics.

Snowball → Glacier Scenario

Snowball cannot send data directly to Glacier. Workflow: Snowball → S3 → Lifecycle Rule → Glacier. This allows you to ingest massive data volumes into S3, then automatically transition to Glacier for long-term archival.

Amazon FSx — Managed Third-Party File Systems

FSx provides fully managed high-performance file systems from third-party vendors (Windows, NetApp, Lustre, OpenZFS) integrated with AWS infrastructure.

FSx for Windows

  • Protocol: SMB (Server Message Block). Windows native file sharing.
  • Scale: Up to 10 Gbps throughput, millions of IOPS, 100s of PB capacity.
  • Features: Fully managed Windows File Server, NTFS, ACLs, user quotas, Microsoft AD integration, DFS namespaces for distributed deployments.
  • Access: Mount from Windows and Linux EC2, on-premises servers (via VPN/Direct Connect).
  • Storage: SSD (latency-sensitive workloads) or HDD (broad spectrum, cost-effective).
  • High availability: Multi-AZ option, daily backups to S3.
  • Exam tip: Choose FSx for Windows when you need SMB protocol, Microsoft AD integration, or migration of existing Windows file server workloads.

FSx for Lustre

  • Protocol: Lustre (parallel distributed file system).
  • Use cases: Machine learning, high-performance computing (HPC), video processing, financial modeling, electronic design automation.
  • Scale: 100s of GBs/sec, millions of IOPS, sub-millisecond latencies.
  • Storage types: SSD (low latency) or HDD (throughput-intensive, cost-effective).
  • Seamless S3 integration: Directly read from S3 as a file system, write compute output back to S3.
  • Deployment options:
    • Scratch: Temporary storage, data not replicated, high burst performance (6x faster, 200 Mbps burst), used for short-term processing, optimizes cost.
    • Persistent: Long-term storage within one AZ, automatic failure recovery within minutes, ideal for sensitive/long-running workloads.
  • Access: On-premises via VPN or Direct Connect; EC2, ECS, EKS.
  • Exam tip: FSx for Lustre with S3 integration is the answer for ML/HPC training pipelines that need to read large datasets and write results.

FSx for NetApp ONTAP

  • Protocol: NFS, SMB, iSCSI (multi-protocol support).
  • Use case: Migrate existing NetApp ONTAP or NAS workloads to AWS without re-architecting applications.
  • Compatibility: Linux, Windows, macOS, VMware Cloud on AWS, Workspaces, AppStream, EC2, ECS, EKS.
  • Features: Automatic storage scaling, snapshots, replication, compression, data deduplication, point-in-time instantaneous cloning for testing.
  • Exam tip: When you see "existing NetApp" or "multi-protocol" requirement (NFS+SMB+iSCSI), FSx for NetApp ONTAP is the fit.

FSx for OpenZFS

  • Protocol: NFS (v3, v4, v4.1, v4.2).
  • Use case: Migrate existing ZFS workloads or need open-source file system with advanced features.
  • Compatibility: Linux, Windows, macOS, VMware Cloud on AWS, Workspaces, AppStream, EC2, ECS, EKS.
  • Performance: Up to 1,000,000 IOPS with 0.5 ms latency.
  • Features: Snapshots, compression, low cost, point-in-time instantaneous cloning.
  • Exam tip: FSx for OpenZFS when "ZFS" is mentioned or when you need NFS with high IOPS and fast cloning.

AWS Storage Gateway — Hybrid Cloud Bridge

Storage Gateway bridges on-premises data centers and AWS cloud, exposing S3 data locally with caching and tiering.

Use Cases

  • Disaster recovery: On-premises backup to AWS.
  • Backup & restore with local caching for fast recovery.
  • Tiered storage: Keep frequently accessed data locally, archive to S3/Glacier.
  • On-premises cache for low-latency file access to cloud data.

Gateway Types

S3 File Gateway

  • Translates on-premises NFS/SMB requests to S3 HTTPS API calls.
  • Protocol: NFS and SMB.
  • Caching: Most recently used data cached locally for low-latency access.
  • S3 Storage Classes: Supports S3 Standard, Standard-IA, One Zone-IA, Intelligent-Tiering.
  • Lifecycle: Use S3 lifecycle policies to transition data to Glacier for long-term archival.
  • Authentication: SMB integrates with Active Directory for user authentication; IAM roles control bucket access.

Volume Gateway

  • Provides block storage via iSCSI protocol, backed by S3 and EBS snapshots.
  • Two modes:
    • Cached Volumes: Recently accessed data cached locally, full dataset in S3. Low latency for hot data.
    • Stored Volumes: Entire dataset on-premises, scheduled backups to S3/EBS snapshots. Useful for disaster recovery without internet dependency.

Tape Gateway (Virtual Tape Library)

  • Replace physical tape backup infrastructure with virtual tapes backed by S3 and Glacier.
  • Protocol: iSCSI.
  • Integrate with existing tape backup software (Backup Exec, NetBackup, Commvault, etc.).
  • Data flow: Backup software → Tape Gateway → virtual tapes → S3 (short-term) → Glacier (long-term archival).
  • Cost: Much cheaper than maintaining physical tapes while retaining long-term retention requirements.

AWS Transfer Family — Managed File Transfer

Fully managed service for file transfer into and out of S3 or EFS using FTP, FTPS, SFTP protocols.

Supported Protocols

Protocol Encryption Port(s) Use Case
FTP None (unencrypted) 20/21 Legacy systems, no encryption requirement
FTPS TLS encryption 990, 21 Secure FTP, compliance requirements
SFTP SSH encryption 22 Modern, secure, widely adopted

Key Features

  • Managed infrastructure: AWS handles scaling, patching, high availability.
  • Pricing: Per provisioned endpoint (per hour) + data transfer (GB).
  • Credentials: Store and manage user credentials within the service or integrate with external identity providers (Microsoft AD, LDAP, Okta, Cognito, custom Lambda).
  • Use cases: File sharing with partners, public dataset distribution, CRM/ERP integrations, data ingestion from third parties.
  • Exam tip: For "legacy FTP/FTPS systems that need to move to AWS," AWS Transfer Family is the answer.

AWS DataSync — Large-Scale Data Transfer

Move large amounts of data between on-premises, other clouds, and AWS storage services at scale with automatic verification and metadata preservation.

Deployment Modes

  • On-premises to AWS: Requires a DataSync agent (deployed as VM or hardware appliance). Syncs to S3, EFS, FSx.
  • AWS to AWS: No agent required. Transfer between different AWS storage services (e.g., S3 to EFS, S3 to FSx).

Key Features

  • Replication tasks can be scheduled (one-time or recurring).
  • File permissions and metadata are preserved (critical for compliance).
  • One agent can use up to 10 Gbps; bandwidth limits can be configured.
  • Automatic verification of data integrity (checksums).
  • Logging: CloudWatch and CloudTrail integration for auditing.

Use Cases

  • Large-scale data migration (100s TB to PB) from on-premises to AWS storage services.
  • Periodic data synchronization between on-prem backup systems and AWS for disaster recovery.
  • Cloud-to-cloud data transfer: Migrating from another cloud provider to AWS.
  • Distributing large datasets: S3 → EFS/FSx across multiple regions or accounts.
  • Continuous data replication for analytics or ML pipelines.
  • Exam tip: When you see "preserve metadata," "large-scale data transfer," or "scheduled replication," DataSync is the answer.

Exam-Style Questions & Answers

  1. Q: Your company has 2 PB of data to migrate from on-premises to AWS. Network bandwidth is 1 Gbps. Estimating timeline and choosing the migration method, what is the best approach?
    A: Over 1 Gbps, 2 PB would take ~2310 days (~6 years). Use AWS Snowball (multiple devices), which takes 2-3 weeks including shipping. This is the only practical solution.
  2. Q: Your machine learning team needs to train models on 500 TB of image data stored in S3. They require ultra-low latency file access and can spin up compute at the edge. Which FSx option is best?
    A: FSx for Lustre with Scratch deployment. It seamlessly mounts S3, provides sub-ms latencies (100s GBs/sec throughput), and optimizes costs with temporary storage for short-term training runs.
  3. Q: Your backup software (Backup Exec) currently backs up to physical tapes. You want to replace tape infrastructure but keep the same backup process. Which AWS service enables this with least operational change?
    A: Storage Gateway Tape Gateway. It presents virtual tapes to existing backup software, stores data in S3, and automatically transitions old tapes to Glacier for archival. No re-architecting needed.
  4. Q: A partner company wants to send files via FTP to your AWS S3 bucket. You need to manage user credentials and audit file transfers. Which service is appropriate?
    A: AWS Transfer Family. It supports FTP/FTPS/SFTP, stores credentials securely, integrates with AD/LDAP/Cognito for authentication, and provides full logging for compliance.
  5. Q: You have 50 TB of data on-premises that needs synced to EFS daily, with file permissions and metadata preserved. What is the best solution?
    A: AWS DataSync with a scheduled daily replication task. It preserves metadata (critical), supports on-prem agents, and automates verification.
  6. Q: Your Windows file servers need to scale to millions of IOPS with multi-AZ HA, AD integration, and DFS namespace support. What do you use?
    A: FSx for Windows. Fully managed, supports SMB, native AD integration, DFS namespaces, multi-AZ with daily S3 backups, and scales to 10 Gbps throughput.
  7. Q: A data preprocessing job needs to run on remote IoT sensors (limited internet) before shipping results to AWS for analytics. Which Snowball variant is suitable?
    A: Snowball Edge (Compute Optimized). Run EC2/Lambda directly on the device to preprocess data locally, then ship the device to AWS for final ingestion.
  8. Q: Your on-premises infrastructure needs to access S3 data with local caching for low latency. Which Storage Gateway type should you deploy?
    A: Storage Gateway File Gateway. It caches recently used S3 data locally via NFS/SMB, supports all S3 storage classes, and integrates with AD for authentication.
  9. Q: You migrated a NetApp ONTAP system to AWS but need to support NFS, SMB, and iSCSI clients simultaneously. What FSx option allows this?
    A: FSx for NetApp ONTAP. It's the only multi-protocol option that supports NFS, SMB, and iSCSI concurrently, perfect for ONTAP workload migration.
  10. Q: Your application needs 1,000,000 IOPS with 0.5 ms latency, NFS protocol, and point-in-time instant cloning for testing. Which FSx variant fits?
    A: FSx for OpenZFS. It delivers exactly those performance characteristics, supports NFS, and offers instantaneous point-in-time cloning for safe testing.

Best Practices & Design Checklist

  • For data migration, assess timeline: If transfer > 1 week, use Snowball. For 100s TB with metadata preservation, use DataSync.
  • Choose protocols based on workload: SMB for Windows, NFS for Linux, iSCSI for block, SFTP for legacy FTP systems.
  • FSx selection: Windows workloads → FSx Windows; ML/HPC → FSx Lustre; existing NetApp → FSx NetApp ONTAP; existing ZFS → FSx OpenZFS.
  • Storage Gateway for hybrid: File Gateway for on-prem S3 caching, Volume Gateway for backup, Tape Gateway to replace physical tapes.
  • Transfer Family for legacy: Use when partners/customers need FTP/FTPS/SFTP ingestion to S3/EFS.
  • Always preserve metadata during migrations using DataSync or native file system features.
  • Plan for high availability: Multi-AZ deployments, automated failover, and backups.

Closing Notes

Storage Extras is critical for exam success. You'll often see scenarios combining multiple services (Snowball → S3 → Lifecycle → Glacier or Storage Gateway + DataSync for hybrid migrations). Practice identifying the right service based on protocol requirements, data volume, and workload type. Protocol knowledge (FTP vs FTPS vs SFTP, SMB vs NFS vs iSCSI) often determines the correct answer in exam questions.
