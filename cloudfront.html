CloudFront & Global Accelerator

Overview

CloudFront is AWS's Content Delivery Network (CDN) that caches content at edge locations to reduce latency and improve user experience. Global Accelerator provides static Anycast IP addresses and routes traffic over the AWS global network to optimal regional endpoints, improving performance, failover and predictability for global applications.

CloudFront — Key Features

  • Caching at Edge Locations: Stores cached copies of content close to users to reduce latency for repeated requests.
  • Origins: S3 (public or private via Origin Access Control), ALB, NLB, EC2, custom HTTP(s) origins and private origins inside a VPC (via ALB/NLB).
  • Cache Control & Policies: Honor origin headers or use CloudFront Cache Policies to control TTLs, query strings, headers and cookies included in the cache key.
  • Security: Integrates with AWS Shield (DDoS) and AWS WAF, supports TLS, Signed URLs/Cookies for restricted content, and field-level protections with Lambda@Edge transformations.
  • Edge Compute: Lambda@Edge (full runtime for request/response transforms) and CloudFront Functions (very fast JS functions for viewer-side manipulations).
  • Geo-Restriction: Allowlist/denylist countries for content delivery.
  • Price Classes: Control which edge locations are used to trade cost vs latency (All / 200 / 100).
  • Invalidation: Force refresh of cached content (supports wildcards). Avoid frequent invalidations by versioning objects.
  • Origin Groups & Failover: Primary/secondary origin configuration for automatic origin failover.

CloudFront — Use Cases

  • Static website and asset delivery (HTML/CSS/JS/images).
  • Accelerating dynamic or API responses when safe to cache (or using partial caching).
  • Streaming media (HLS/DASH) and large-file downloads.
  • Protecting origins with WAF and Shield; reducing origin load.
  • Edge-based request/response manipulation (image resizing, A/B tests) using Lambda@Edge / CloudFront Functions.

CloudFront — Best Practices

  • Minimize Cache Key Variance: Only include necessary headers, cookies and query strings to boost cache hit ratio.
    • - Use cache-control headers and fine-tune TTLs at origin or with CloudFront policies.
  • Origin Shield: Use Origin Shield (regional caching layer) to consolidate origin traffic and reduce origin load.
  • Versioned Object Names: Fingerprint assets (e.g., app.v1.js → app.v1.abc123.js) to avoid invalidations.
  • Secure Private S3 Origins: Use Origin Access Control (OAC) or Origin Access Identity (OAI) instead of public buckets; issue Signed URLs/Cookies for restricted downloads.
  • Use CloudFront + WAF + Shield: Protect from layer 7 attacks at the edge and minimize origin exposure.
  • Edge Compute Choice: Use CloudFront Functions for small, fast manipulations; use Lambda@Edge for heavier logic requiring libraries or more runtime.
  • Monitoring: Use CloudFront access logs, CloudWatch metrics (cache hit ratio, 4xx/5xx) and real-user monitoring to tune performance.

CloudFront — Exam Tips

    - Choose CloudFront for caching, edge security (WAF), signed URLs, and edge transformation.
    - Prefer versioned object names over frequent invalidations to save cost and accelerate deployments.
    - CloudFront + S3 static site: use CloudFront + OAC and ACM certificate for HTTPS (S3 website endpoint alone does not provide HTTPS for custom domains).
    - Price classes reduce edge locations to lower cost at the expense of latency in excluded regions.

Global Accelerator — Key Features

  • Anycast Static IPs: Provides two static Anycast IP addresses (or BYO IPs with Elastic IP) that remain constant for clients.
  • AWS Global Network: Client traffic enters the nearest AWS edge and traverses the AWS backbone to the best regional endpoint, reducing internet variability.
  • Endpoint Types: ALB, NLB, EC2 instances, Elastic IPs (and private ALB/NLB endpoints via regional endpoints).
  • Health Checks & Traffic Control: Built-in health checks, automatic failover and traffic weighting for progressive rollouts (blue/green or canary).
  • Protocol Support: TCP & UDP friendly — ideal for non-HTTP use cases.
  • Security & Simplicity: Whitelist two static IPs on client firewalls; integrates with Shield for DDoS protection.

Global Accelerator — Use Cases

  • Real-time, latency-sensitive applications (gaming, VoIP, IoT ingestion).
  • Applications that require static IPs for firewall whitelisting.
  • Multi-region active/active or active/passive deployments needing fast failover.
  • Non-HTTP protocols (UDP/TCP) where CloudFront is not suitable.
  • When deterministic routing and consistent performance are required.

Global Accelerator — Best Practices

  • Combine with Regional Load Balancing: Use Global Accelerator fronting ALBs/NLBs in multiple regions to achieve global availability and fast failover.
  • Use Health Checks & Weights: Configure endpoint groups with health checks and weights for seamless traffic shifting.
  • Whitelist Minimal IPs: Use the two static Accelerator IPs on customer firewalls to reduce complexity.
  • Cost vs Benefit: Accelerator adds hourly and data charges — justify its use for static IP needs, non-HTTP traffic or critical global performance SLAs.
  • Monitoring: Track CloudWatch metrics (client latency, endpoint health, traffic shift) to verify behavior and failover.

Global Accelerator — Exam Tips

    - Pick Global Accelerator when the question mentions static IPs, non-HTTP protocols, or the need for deterministic global routing.
    - Global Accelerator routes traffic into the AWS backbone (lowers variability) and is ideal for fast regional failover and predictable network performance.

CloudFront vs Global Accelerator — Decision Guide

  • CloudFront: Best for caching HTTP(s) content, edge transformations, signed URLs, and integration with WAF for application-layer protection.
  • Global Accelerator: Best for static Anycast IPs, optimized routing for TCP/UDP, predictable performance and simplified client firewall rules.
  • Combined Architectures: Use CloudFront for static & HTTP caching and Global Accelerator for application endpoints requiring static IPs or consistent network routing. Both can coexist where architectures need caching plus global routing.

Architecture Patterns & Examples

  • Secure Static Website: S3 (private) + CloudFront (OAC) + ACM TLS + WAF = HTTPS + edge security + origin protection.
  • Web APIs with caching: CloudFront in front of ALB; use path-based cache behaviors and origin shielding.
  • Global Low-Latency App: Global Accelerator fronting NLBs/ALBs in multiple regions for real-time traffic; use CloudFront for static assets.
  • Games or VoIP: Global Accelerator + NLB/EC2 for UDP/TCP endpoints, health checks and traffic weighting for regional failover.

Performance & Cost Considerations

  • CloudFront pricing varies by edge location; Price Classes can reduce cost by excluding expensive regions at a latency tradeoff.
  • CloudFront costs include data transfer out, requests and optional Lambda@Edge/Function charges; invalidations beyond the free quota are billed.
  • Global Accelerator has an hourly charge per accelerator plus data transfer charges over the AWS backbone; it can reduce client-side variability but increases bill.
  • Tune CloudFront cache hit ratio (cache keys, TTLs, compression) to reduce origin costs and improve latency.

Troubleshooting & Monitoring

  • CloudFront: Use access logs, CloudFront metrics (Requests, BytesDownloaded, 4xx/5xx errors), and real-user metrics to find cache misses, origin errors, or misconfigured behaviors.
  • Global Accelerator: Check endpoint group health, CloudWatch metrics for latency and traffic, and ensure endpoint weights and health checks are correctly set.
  • Common Issues: HTTPS certificate errors (ACM region mismatch), origin authentication problems (OAC/OAI), excessive invalidations, and unexpected cache misses due to wide cache key variance.

Common Exam Scenarios & Answers

  • Scenario: "Serve global static assets over HTTPS and protect the origin from spikes."
    Answer: S3 (private) + CloudFront (OAC) + ACM certificate + WAF + Origin Shield.
  • Scenario: "Game servers use UDP and need low, consistent latency globally and a static IP for firewall rules."
    Answer: Global Accelerator fronting NLB/EC2 (UDP) endpoints; enable health checks and traffic weights.
  • Scenario: "Single pair of IPs to whitelist at customer firewalls and fast failover across regions."
    Answer: Global Accelerator with ALB/NLB endpoints in multiple regions; optionally use CloudFront for cached static content.
  • Scenario: "Transform images at the edge and remove PII before serving."
    Answer: CloudFront + Lambda@Edge (or CloudFront Functions for light transforms) to redact/transform on read.

Quick CLI / Practical Notes

  • Create a CloudFront distribution for an S3 origin and configure Origin Access Control to restrict bucket access to CloudFront only.
  • Use fingerprinted filenames (versioning) to avoid invalidations when deploying new assets.
  • For Global Accelerator, create endpoint groups per region and set health checks/weights for controlled traffic shifts.
  • When using CloudFront with private ALB/EC2 origins in a VPC, set up an ALB with appropriate security groups and OAC or signed origin requests.

Final Best-Practices Summary

  • Use CloudFront for HTTP(s) caching, edge compute, and integration with WAF/Shield.
  • Use Global Accelerator for static IPs, consistent routing over AWS backbone, non-HTTP protocols, and fast global failover.
  • Combine CloudFront & Global Accelerator when you need both caching and static IP/global routing — but balance cost vs benefit.
  • Secure origins (OAC/OAI), use TLS, monitor logs and metrics, and prefer versioned asset names to reduce invalidations.
